Android unit testing practice

发表于 2023-02-20 更新于 2023-02-22 分类于 Android Testing深入浅出

basic concepts

According to Google’s official recommendations, the Android testing system should refer to the testing pyramid architecture (as shown in the figure below), and the App should include three types of tests (i.e. small, medium and large tests)：

  • Small tests are unit tests that verify the behavior of an application, one class at a time.
  • Medium tests refer to integration tests that verify interactions between stack levels within a module or between related modules.
  • Large-scale testing refers to end-to-end testing, which is used to verify the user UI click operation process across multiple modules of the application.

Step by step up the pyramid, from small tests to large tests, the fidelity of various tests (the real experience of users) increases step by step, but the execution time and workload required for maintenance and debugging work also increase step by step. Therefore, we should write more unit tests than integration tests, and more integration tests than end-to-end tests. While the proportion of each type of test may vary depending on your application's use cases, we generally recommend the following proportions: 70% for small tests, 20% for medium tests, and 10% for large tests.

Today we are mainly talking about small tests, which account for 70%, also called unit tests.

Introduction to testing framework

basic framework：JUnit

https://junit.org/junit4/
JUnit is a unit testing framework for the Java language. Junit testing is programmer testing, the so-called white-box testing, because we know how (How) the software under test completes the function and what function it completes (What). Junit is a set of frameworks. If you inherit the TestCase class, you can use Junit for automatic testing. Most Java development environments have integrated JUnit as a unit testing tool.

mock框架

Mock通常是指，在测试一个对象A时，我们构造一些假的对象来模拟与A之间的交互，而这些Mock对象的行为是我们事先设定且符合预期。通过这些Mock对象来测试A在正常逻辑，异常逻辑或压力情况下工作是否正常。

Robolectric框架

Robolectric通过实现一套JVM能运行的Android代码，然后在Junit test运行的时候去截取android相关的代码调用，然后转到Robolectric内部实现的代码去执行这个调用的过程。不用依赖真实的 Android 环境中运行（模拟器或者真机）
Robolectric主要适用于UI的测试，比如Activity，Fragment，一些页面操作的测试场景，采用Shadow的方式对Android中的组件进行模拟测试，从而实现Android单元测试Robolectric正好弥补了Mockito的不足，两者结合使用是最完美的，即如果想要测试依赖了Android FrameWork相关的代码，就需要使用Robolectric的能力。

Step1:配置测试环境

根据执行环境组织整理测试目录,Android Studio 中的典型项目包含两个用于放置测试的目录。请按以下方式组织整理您的测试：

  • androidTest 目录应包含在真实或虚拟设备上运行的测试。此类测试包括集成测试、端到端测试，以及仅靠 JVM 无法完成应用功能验证的其他测试。
  • test 目录应包含在本地计算机上运行的测试，如单元测试。

考虑在不同类型的设备上运行测试的利弊,在设备上运行测试时，您可以从以下类型中进行选择：

  • 真实设备
  • 虚拟设备（如 Android Studio 中的模拟器）
  • 模拟设备（如 Robolectric）

Real devices provide the highest fidelity, but also take the most time to run the tests. Analog devices, on the other hand, offer higher test speeds at the expense of lower fidelity. However, platform improvements in binary resources and realistic looping procedures allow simulated devices to produce more realistic results.
Virtual devices balance fidelity and speed. When we use virtual appliances for testing, snapshots can be used to minimize setup time between tests.

Unit test writing experience summary

TDD(Test Driven Development):test driven development

TDD is a core practice and technology in agile development and a design methodology. The principle of TDD is to write unit test case code before developing functional code. The test code determines what product code needs to be written. TDD is a core practice of XP (Extreme Programming). Its main promoter was Kent Beck.
