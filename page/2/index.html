Android Gradle Best Practice Series 8: Setting up CI (Integrated Development Environment)

Posted in 2016-05-02 updated on 2022-04-08 Classification于 Android Gradle best practices series

持续集成（CI） It is a development practice that requires developers in the team to regularly integrate code compilation and testing work, usually multiple times a day, and each push to the main code repository needs to be verified through automatic builds. This practice helps to detect problems as quickly as possible, thus speeding up development and improving the quality of the code. David Martin Fowler wrote an article on this topic that explains the concepts and describesContinuous integration best practices

There are several ways to set up CI for Android. The most widely used are Jenkins, TeamCity and Travis CI. Jenkins has the largest ecosystem, with around 1,000 plugins available. Jenkins itself is an open source project and still has many contributors. TeamCity is a product of JetBrains. We should all be familiar with JetBrains. JetBrains also created IntelliJ IDEA (Android Studio is based on this development). Travis CI is relatively new and is mainly used for open source projects.

In this chapter, we will learn about these CI systems and how to make Gradle work on these systems. At the end of the chapter, you'll also learn some Gradle tricks to make automating CI easier.

Android Gradle Best Practices Series 5: Multi-module build

Posted in 2016-04-20 updated on 2022-04-08 Classification于 Android Gradle best practices series

Android Studio can create modules not only for apps and libraries, but also for Android Wear, Android TV, Google App Engine, and more. All these modules can be managed together in a single project. For example, we might need to create an app that uses Google Cloud Endpoints as a backend and includes integration with Android Wear. In this case, you need to have a project with three different modules: one for the mobile app, one for the backend, and one for the Android Wear integration. Understanding the structure and construction of multi-module projects can significantly speed up our development efficiency and shorten the development cycle.

The documentation for both Gradle and the Gradle Android plugin uses the terminology of multiproject builds. However, in Android Studio, there is a difference between modules and projects. A module refers to a single application: an Android application or a Google App Engine backend, for example. A project is a collection of all modules. In this book, we use the terms module and project in the same way as IDEs to avoid confusion. Please keep this in mind when reading this article.

Android Gradle Best Practices Series 4: Creating Build Variants

Posted in 2016-04-17 updated on 2022-04-08 Classification于 Android Gradle best practices series

When we develop an application, we usually face the need to release different versions. Let’s take two common scenarios. Scenario 1: We are developing new features. After the development is completed, we need to release a test version and submit it to QA testers. After passing the test, the online version will be released. At this time, the server interface domain names of the offline version and the test version are different or have different API interfaces; Scenario 2: Our app needs to release a free version and a paid version. The paid version will have Higher usage rights. For the above two situations, we need to release four apk, free QA version, free online version, paid QA version, paid online version. If hard-coded in the code, the project will be different. Very complex. In response to this situation, gradle has proposed a method to solve this problem. For the QA version or online version, you can configure thebuild type，Androidstudio is configured with two types: Debug and Release by default. For the paid version or the free version, you can configure Build flavors. The combination of these two types is calledbuild variant

Build types

Gradle's build type in Android is used to handle what type of app or library should be built. In this configuration, we can define what the application's package name is, whether to automatically remove unreferenced resources, whether to enable obfuscation, etc. .
